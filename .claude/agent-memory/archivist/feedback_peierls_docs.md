Private functions (prefixed with `_`) in modules not listed in autodoc will NOT resolve as `:func:` or `:class:` Sphinx references. Use inline code (double backticks) instead, optionally with `:mod:` pointing to the parent module if it IS in autodoc.

**Why:** The `peierls_slab` / `peierls_cylinder` modules are not in `docs/api/derivations.rst` automodule list, so `:func:orpheus.derivations.peierls_cylinder._optical_depth_along_ray` produces a warning. `:mod:orpheus.derivations.peierls_cylinder` still works if the module is importable.

**How to apply:** Before using `:func:`, `:class:`, or `:meth:` cross-refs, check `docs/api/` for the relevant automodule directive. If absent, use backtick-quoted inline code. Top-level `@dataclass` classes and exported functions in the module CAN be referenced via `:class:` / `:func:` even without automodule, as long as the module is on sys.path (autodoc.ext resolves them).

The `\Sigt{}` macro takes exactly one argument (possibly empty). Writing `\Sigt{,i}` produces `\Sigma_{t,,i}` (double comma). Correct form: `\Sigt{i}`.

**Title hierarchy** in `docs/theory/collision_probability.rst` is `====` (h1) → `----` (h2) → `^^^^` (h3). Do NOT use `~~~~` for h3 — the file never uses that marker at any level, so introducing it triggers "Inconsistent title style" ERROR.

**vv-status: documented comments** work like this: add a plain RST comment `.. vv-status: <label> documented` immediately after the `.. math::` block that carries the label. Sphinx ignores it (not a registered directive); the V&V audit picks it up and drops the label from the orphan-equation gate. Use this when you introduce new `:label:` blocks that existing tests hit but via a different decorator name (e.g., cylinder labels when the cylinder tests mark `verifies("peierls-equation")` for the slab label). Do NOT use it to paper over genuine V&V gaps.

**Citations in collision_probability.rst** are defined as `.. [Key] ...` bibliography entries near the end. Every added `.. [Key]` MUST be cited at least once via `[Key]_` in the narrative, or Sphinx emits `WARNING: Citation [Key] is not referenced`. Adding only narrative text like "Sanchez 1982" without the `[Sanchez1982]_` wrapper leaves a silent gap — the bibliography entry becomes orphan.

**Sphinx citations DO resolve cross-document** (retested 2026-04-18 during Phase H.1 — `[Woodcock1965]_` in peierls_unified.rst cleanly resolved to the definition in monte_carlo.rst with no warning, and the HTML output shows a correct cross-page link). The earlier note about "strictly per-document" was wrong or reflected a config that has since been fixed. Practical rules: (a) **do NOT duplicate** `.. [Key]` definitions across files — that produces a "duplicate citation" warning; (b) define each `[Key]` in exactly one file (typically the page that introduces the topic), and cite it freely from other files. The only remaining failure mode is `[Key]_` inside a `.. math::` block, which Sphinx treats as literal math text; always put citation refs in surrounding prose.

**csv-table breaks on commas inside LaTeX.** A directive like `.. csv-table::` with `:math:\`[0, 2\pi)\`` in a cell field treats the internal comma as a column separator, raising "widths do not match the number of columns". Always use `.. list-table::` for any row containing LaTeX with commas (most math rows).

**Title overline must match title length in CODE POINTS (not bytes, not display cells).** UTF-8 characters like "ö" are 1 code point but 2 bytes; em-dash "—" is 1 code point but 3 bytes. Docutils counts code points via Python `len(str)`. `wc -c` counts bytes (wrong); `awk '{print length}'` counts bytes on most implementations (wrong); use `python3 -c "print(len('...'))"` to count exactly what docutils sees. Common trap: subsection headings with "Nyström" + "—" where hand-counted tildes look right but are short. Re-confirmed 2026-04-18 in Phase H.1 where four subsection headings each had an underline 1 char too short (the em-dash accounted for the phantom char).

**Sphinx citations inside math blocks don't resolve.** `.. math::` content with `[Key]_` is treated as literal math text, not as a citation reference. The `[Key]` bib entry will then emit "citation not referenced" even though the pattern appears in the source. Always put `[Key]_` in the surrounding prose, not inside `.. math::`.

**Toctree placement editorial decision.** In this project, new theory pages go in the top-level `docs/index.rst` Theory & Derivations toctree, NOT in `transport_methods.rst`. The user's first instinct is typically to keep `transport_methods` a curated cluster of the four primary transport solvers (SN/CP/MoC/MC) and promote cross-cutting theory pages to top-level. User intervention observed 2026-04-18: they moved peierls_unified out of transport_methods.rst and back into the top-level toctree.

**Simple-table column widths are a trap when cells contain LaTeX math.** `\mathrm{Ki}_1` renders as 6 characters visually but occupies 21 bytes in the .rst source; docutils measures the column by the SOURCE width, not the rendered width, so a simple table with `===` column separators that "look" aligned with the cell text will ERROR ("Malformed table") as soon as a math cell exceeds the separator length. Observed 2026-04-18 while writing the sphere Peierls section: three simple tables each broke on a cylinder-row math cell. **Default to `.. list-table::` for any table containing `:math:` cells.** The structure-first form (explicit `* -` rows) is immune to source-width counting.

**Run `python -m sphinx -W -b html docs docs/_build/html` after EVERY substantial doc edit.** Malformed tables surface as ERROR (not WARNING), and a single malformed table silently loses the entire table in the HTML output even though the build completes. Do not rely on "build finished" as correctness — read the WARNING/ERROR block.

**Forward-referencing code modules in `.. seealso::` is fine** when the code is coming in a later phase. The Sphinx build emits no warning for `:mod:orpheus.derivations.cp_geometry` pointing at a not-yet-existing module (autodoc only warns when you try to *document* something that doesn't exist). Nexus will surface it as an unresolved reference; the Phase B.2 commit that creates `cp_geometry.py` will resolve it. This is the correct pattern when theory lands before code (Cardinal Rule 2 — docs first, so code has a clear target). Observed 2026-04-18 writing peierls_unified.rst §§11-17 (Phase B.1).

**Cross-document `:eq:` references work unconditionally** even when they cross from `peierls_unified.rst` to `collision_probability.rst`. Unlike `.. [Key]_` citations (which are strictly per-document), `:eq:\`label\`` and `:ref:\`target\`` resolve globally across the Sphinx project. This is what makes a "unified theory" page viable: it can cite the existing per-geometry derivations without re-deriving them.

**Moment-form / closed-form-recursion documentation pattern.** Documenting a "closed-form replaces quadrature" architecture needs five distinct subsections to give a future session full context: (1) the unified statement showing the contraction structure ⟨coeffs, moments⟩, (2) the closed-form recursions per kernel with intermediate IBP steps shown explicitly (cite the LewisMiller App. C / Hébert §3.x / Stamm'ler Ch. 4-6 trio for slab/cylinder; A&S §5.1 / §11.2 underlying), (3) the algorithmic specialization step-by-step (ray walk, segment splitting, Vandermonde solve, accumulation), (4) the failed-approach diagnosis with empirical convergence tables AND the theoretical exactness-class explanation (e.g., "all Laguerre flavours are spectral only for polynomial-in-v"), (5) numerical evidence in a four-tier ladder (term gates → moment recursion gates → segment-difference gates → element-wise K-matrix gates → power-iteration eigenvalue gates). The diagnostic-script tables go in subsection (4); the implementation-test tables go in subsection (5). Both are essential — the first prevents re-attempting the failed approach, the second provides confidence floor.

**Cross-document `[Key]_` references duplicate-warning trap (still active 2026-04-19).** The existing per-page bibliographies in `peierls_unified.rst` and `collision_probability.rst` have ~6 duplicate citation entries (Sanchez1982, Hebert2020, Stamm1983, BellGlasstone1970, Carlvik1966, Atkinson1997). New sections that cite these keys can do so freely (the citations DO resolve cross-document); just **do not add a new `.. [Key]` definition** in the new section's References subsection — refer to the existing definitions in the page-end References block. Verified clean build with zero new warnings using normal `sphinx-build` (the `-W` flag promotes the pre-existing duplicate-citation warnings to errors but they were already there).

**`:ref:` to a label not adjacent to a section title needs explicit text (2026-04-28).** A `.. _label:` placed *after* a section heading underline (e.g. between underline and the first paragraph) is a *generic anchor*, not a *section reference target*: Sphinx will warn "Failed to create a cross reference. A title or caption not found: 'label'" for any unnamed `:ref:\`label\``. Two fixes: (a) move the label to immediately *before* the title underline, or (b) provide explicit link text via `:ref:\`§22.7 <label>\``. Surfaced 2026-04-28 when the `gauss_legendre_visibility_cone` docstring's `:ref:\`section-22-coordinate-transforms\`` failed because that label sits between the title and the first body paragraph. Don't rely on auto-titled refs unless the label is *directly above* the title.

**Subsubsection underline length re-trap (2026-04-28).** Even a routine `~~~` underline trips the docutils width counter when the subsubsection title contains punctuation that nudges the length: `Where this singularity comes from in ORPHEUS` is exactly 44 code points and needs ≥44 tildes; `Spectral convergence and the Bernstein-ellipse caveat` is 53 code points and needs ≥53. Always count with `python3 -c "print(len('...'))"` after typing a new subsubsection, not by eyeballing the underline against the title text.
